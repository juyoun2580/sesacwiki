# DECISION_TREE.md

새 기능을 추가할 때 **어느 파일을 수정해야 하는지** 판단하기 위한 흐름도다. CLAUDE.md의 Claude Workflow(구조 분석 → 기존 컴포넌트 확인 → 기존 CSS/JS 재사용 확인 → 신규 파일 필요성 검토) 순서를 팀원이 직접 따라갈 수 있도록 구체화했다.

관련 문서: [COMPONENTS.md](COMPONENTS.md) · [PAGES.md](PAGES.md) · [JSON_GUIDE.md](JSON_GUIDE.md) · [WORK_ORDER.md](WORK_ORDER.md)

---

## 0. 먼저: 무엇을 추가/변경하려 하는가?

```
무엇을 추가하려 하는가?
├── A. 화면에 보이는 UI 요소 (버튼, 카드, 뱃지, 색상 등)         → 1번
├── B. 페이지 콘텐츠/섹션/레이아웃 구조                          → 2번
├── C. 클릭/입력 등 동작(JS 인터랙션)                            → 3번
└── D. 목록/텍스트 등 데이터                                     → 4번
```

## 1. UI 요소(버튼/카드/뱃지/색상 등)를 추가하고 싶다

예: "버튼을 추가하고 싶다"

```
1-1. COMPONENTS.md에 이미 있는 컴포넌트인가?
│
├── 예 (예: 이미 .btn, .tag, .card 등이 존재)
│     └──▶ 새 CSS를 작성하지 않는다.
│          기존 클래스 + 기존 modifier를 그대로 사용한다.
│          예: <button class="btn btn--primary">
│          (끝)
│
└── 아니오 (기존 컴포넌트로 표현이 안 됨)
      │
      1-2. 기존 컴포넌트의 modifier 하나 추가로 해결되는가?
      │    예: "버튼에 위험(danger) 색상이 하나 더 필요하다" 수준
      │
      ├── 예 ──▶ components.css 수정?
      │           └──▶ 예. components.css에 modifier 추가
      │                (공통 담당자가 작업 또는 승인)
      │                (끝)
      │
      └── 아니오 (완전히 새로운 형태의 UI)
            │
            1-3. 2개 이상의 페이지에서 재사용될 가능성이 있는가?
            │
            ├── 예 ──▶ components.css에 신규 컴포넌트 추가
            │           (반드시 공통 담당자 승인 필요, COMPONENTS.md
            │            "새로운 Component를 만들 수 있는 조건" 4가지 확인)
            │           (끝)
            │
            └── 아니오 (지금 만드는 페이지 1곳에서만 쓰인다)
                  └──▶ 아니다.
                       └──▶ pages/{담당페이지}.css 수정
                            (해당 페이지 담당 팀이 직접 작업)
                            (끝)
```

**추가 규칙**: 새 색상이 필요하면 항상 `variables.css`에 토큰을 먼저 추가한다 (하드코딩 색상값 금지, CLAUDE.md CSS Rules). 토큰 추가는 공통 담당자 소관이다.

## 2. 페이지 콘텐츠/섹션/레이아웃을 추가하고 싶다

예: "취업핸드북에 새로운 카드 섹션을 추가하고 싶다" / "완전히 새로운 메뉴(페이지)를 만들고 싶다"

```
2-1. 기존 9개 페이지(home/wiki/detail/exam/quiz/job/my/mywords/myfav)
     중 하나에 속하는 콘텐츠인가?
│
├── 예 ──▶ index.html 안의 해당 <section id="...">...</section>
│           블록 내부에서만 작업한다.
│           (해당 페이지 담당 팀, 다른 section·header·nav는 손대지 않음)
│           레이아웃 뼈대(.shell/.sidebar/.side-panel)가 필요하면
│           layout.css를 참고해 재사용하고, 새 레이아웃 클래스를
│           만들지 않는다.
│           (끝)
│
└── 아니오 (완전히 새로운 페이지 = nav에 새 메뉴가 필요함)
      └──▶ 공통 담당자와 먼저 협의한다.
           1) nav__item 추가 (공통 담당자, layout.css/index.html <nav>)
           2) 신규 <section id="새페이지">.page 골격 생성 (공통 담당자)
           3) PAGE_NAV 매핑 추가 (공통 담당자, assets/js/app.js)
           4) 이후 신규 pages/{새페이지}.css 및 담당자 배정
           (페이지 담당 팀이 임의로 nav나 신규 section 골격을 먼저
            만들지 않는다 — 충돌 방지)
           (끝)
```

## 3. 클릭/입력 등 동작(JS 인터랙션)을 추가하고 싶다

예: "즐겨찾기 별을 눌렀을 때 동작을 바꾸고 싶다" / "새로운 페이지 전용 팝업을 만들고 싶다"

```
3-1. 여러 페이지에서 공통으로 쓰이는 동작인가?
     (예: 즐겨찾기 토글, 필터탭, 정렬, 사이드바, 모달, 토스트, 페이지 전환)
│
├── 예 ──▶ assets/js/ui.js / assets/js/modal.js / assets/js/toast.js / assets/js/app.js 수정
│           (공통 담당자 전담. 페이지 팀은 이 파일들을 직접 수정하지
│            않고, 필요한 변경 사항을 공통 담당자에게 요청한다)
│           (끝)
│
└── 아니오 (특정 페이지 1곳에서만 필요한 동작)
      │
      3-2. 이미 그 페이지의 전용 JS 파일이 있는가?
      │    (wiki/detail → assets/js/pages/highlight.js, exam/quiz → assets/js/pages/quiz.js)
      │
      ├── 예 ──▶ 해당 파일에 기능을 추가한다 (해당 페이지 담당 팀)
      │           (끝)
      │
      └── 아니오 (home/job/my/mywords/myfav처럼 전용 JS가 아직 없음)
            └──▶ assets/js/pages/{페이지그룹}.js 신규 생성
                 (해당 페이지 담당 팀. data-action 속성 위임 패턴을
                  따르고, index.html 최하단 <script> 태그 목록에
                  추가하는 부분만 공통 담당자에게 요청한다)
                 (끝)
```

## 4. 목록/텍스트 등 데이터를 추가하고 싶다

예: "위키 문서를 하나 더 추가하고 싶다" / "모의고사 문제를 추가하고 싶다"

```
4-1. 기존 assets/data/*.json 파일(JSON_GUIDE.md 스키마)의 배열/필드에 항목을
     추가하는 것으로 표현되는가?
│
├── 예 ──▶ 해당 JSON 파일에 항목 추가
│           (파일 소유 팀이 직접 작업, JSON_GUIDE.md의 필드 형식을 따른다.
│            아직 JSON으로 분리되지 않았다면 index.html의 해당
│            <section> 안에 정적 마크업으로 추가해도 무방하다 —
│            단 반복되는 리스트 항목이 5개를 넘으면 JSON 분리를
│            우선 검토한다)
│
└── 아니오 (기존 스키마로 표현이 안 되는 새로운 종류의 데이터)
      └──▶ 새 JSON 파일이 필요한가?
           1) JSON_GUIDE.md에 새 파일/새 필드 스키마를 제안한다
           2) 다른 페이지와 중복되는 정보가 아닌지 확인한다
              (예: myfav처럼 다른 도메인 데이터의 부분집합이면
               새 파일을 만들지 않고 isFavorite 필드로 표현)
           3) 공통 담당자 승인 후 파일 생성 + JSON_GUIDE.md 갱신
           (끝)
```

## 요약 체크리스트

새 작업을 시작하기 전에 아래 순서로 확인한다 (CLAUDE.md Claude Workflow와 동일):

1. [COMPONENTS.md](COMPONENTS.md) — 재사용 가능한 컴포넌트가 있는가?
2. [PAGES.md](PAGES.md) — 이 작업이 내 담당 페이지/파일 범위 안에 있는가?
3. [JSON_GUIDE.md](JSON_GUIDE.md) — 이미 정의된 데이터 스키마로 표현되는가?
4. 셋 다 "아니오"라면 → 신규 파일/컴포넌트가 정말 필요한지 검토하고, **1개 페이지 전용이면 직접, 2개 페이지 이상 공유되면 공통 담당자 승인**을 받는다.
5. 파일 수정 권한은 항상 [WORK_ORDER.md](WORK_ORDER.md)의 "수정 가능한 파일" 범위를 따른다.
